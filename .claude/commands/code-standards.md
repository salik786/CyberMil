## TypeScript
- `strict: true` is enforced ‚Äî no `any`, no `as unknown as X` casts without a comment explaining why
- Derive types from the auto-generated `lib/types/database.types.ts` (generated by `supabase gen types`) ‚Äî never manually duplicate DB types
- Use `type` not `interface` for plain data shapes; use `interface` only when extending
- Narrow union types explicitly ‚Äî never assume a type is safe without checking

## Next.js App Router
- Default to **Server Components** ‚Äî only add `"use client"` when the component uses hooks, browser APIs, or event handlers
- Never import `lib/supabase/server.ts` inside a `"use client"` file
- Mutations go in **Server Actions** (`actions.ts` co-located with the page), not in `useEffect` or inline fetch calls
- Use `revalidatePath` / `revalidateTag` after mutations ‚Äî never do manual client-side state refresh of server data
- Route handlers (`app/api/`) are only for webhooks and third-party callbacks ‚Äî all other data access goes through Server Components or Server Actions

## Supabase
- Always check for `error` after every Supabase call ‚Äî never destructure only `data` and ignore `error`
- Never use `supabase.from(...).select('*')` ‚Äî always specify the columns you need
- RLS handles authorization at the DB layer ‚Äî do not duplicate role checks in application code that already have RLS policies
- Use `SUPABASE_SERVICE_ROLE_KEY` only in Server Actions or Route Handlers that intentionally bypass RLS (admin operations); document why

## Error Handling
- User-facing errors must show a human-readable message ‚Äî never expose Postgres error codes or stack traces to the UI
- Server Actions must return a typed result `{ data, error }` ‚Äî never `throw` from a Server Action
- Use `notFound()` from `next/navigation` for missing resources, not a custom 404 component

## File & Naming Conventions
- Files: `kebab-case.tsx` for components, `kebab-case.ts` for utilities
- Components: `PascalCase` named exports ‚Äî no default exports except for Next.js page/layout files
- Co-locate: `actions.ts`, `loading.tsx`, `error.tsx` live next to the `page.tsx` they belong to
